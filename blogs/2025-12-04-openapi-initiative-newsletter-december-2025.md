<h1 id="openapi-initiative-newsletter-december-2025">OpenAPI Initiative Newsletter – December 2025</h1>

<h3 id="initiative-news">Initiative News</h3>
<p>There’s been a enthusiastic reception to our OpenAPI Specification v3.2 release, with a great deal of coverage and interest from the community in the new features and capabilities. As a reminder, v3.2 provides a host of changes such as the refactored and improved <a href="https://spec.openapis.org/oas/latest.html#tag-object">Tag Object</a>, support for the <code>QUERY</code> HTTP Method, support for sequential and streaming data protocols, and additional Security Scheme features like OAuth 2.0 Device Authorization Flow. You can find out more in our <a href="https://www.openapis.org/blog/2025/09/23/announcing-openapi-v3-2">blog post</a>), which includes links to key resource to help with upgrading to v3.2. Our <a href="#ecosystem-spotlight">Ecosystem Spotlight</a> also provides resources from the community on what the v3.2 upgrade means.</p>
<p>Completing v3.2 means we are now looking at what could appear in our next version and beyond, and the team is ramping up for further releases, with a view to v3.3 that covers improvements to Security Schemes and greater integration with MCP and AI protocols more generally. If you want to find out more, please checkout the Discussions on the OpenAPI Specification <a href="https://github.com/OAI/OpenAPI-Specification/discussions">repository</a>.</p>
<p>Work has also started for an Arazzo Specification v1.1.0 release, with perhaps the most significant change being the addition of support for AsyncAPI. AsyncAPI support will provide the means to describe workflows for both HTTP-based and message-orientated APIs, providing a significant uplift to how Arazzo can describe sequences of API operations with different architectural styles and provide improved capabilities for calling sub-workflows. Other planned enhancements include improvements to JSONPath and XPath support, and a number of fixes identified since v1.0.1. You can find out more about the plans for v1.1.0 <a href="https://github.com/OAI/Arazzo-Specification/discussions/379">here</a>.</p>
<p>Overlay is also gearing up for a v1.1.0 release, with several features in the frame such as clarifications to format interoperability and a new Parameter Object in the frame. If you want to contribute or simply find out more about what’s going on, head over to the Overlay Specification <a href="https://github.com/OAI/Overlay-Specification">repository</a>.</p>
<h3 id="ecosystem-spotlight">Ecosystem Spotlight</h3>
<p>We’ve already mentioned the release of v3.2 and we’ve had a great feedback from the community. Here’s a few samples of what folks are saying.</p>
<ul>
<li>
<p><a href="https://www.linkedin.com/in/daveshanley/">Dave Shanley</a> brought together a really comprehensive <a href="https://quobix.com/articles/openapi-3.2/">overview</a> of all the new features of v3.2 (“<em>I love that new feature smell </em>”) with plenty of snippets showing exactly how to implement them.</p>
</li>
<li>
<p><a href="https://www.linkedin.com/in/anton-okolelov/">Anton Okolelov</a> also <a href="https://medium.com/@okoanton/openapi-3-2-0-evolving-with-modern-api-patterns-9977972e9381">dug deep</a> in his overview, describing streaming support as “<em>a long overdue addition</em>”. He summarizes the value of v3.2 in this way: “<em>While no specification can anticipate every future pattern, OpenAPI 3.2.0 demonstrates a willingness to adapt to observed practices rather than dictating them. That’s a healthy approach for any evolving standard.</em>”</p>
</li>
<li>
<p><a href="https://www.linkedin.com/in/zaiddabaeen/">Zaid Daba’een</a> describes the update as focusing “<em>…on the real pain points that show up when you scale: messy docs, patchy auth support, and unclear streaming behavior.</em>”</p>
</li>
</ul>
<p>For the contributors involved in the creation of v3.2, of which there was the greatest number yet in OpenAPI Specification versions, this level of validation is vital for the ongoing development of the OpenAPI Initiative Specifications. Getting great feedback &#8211; good and bad &#8211; which puts the new features in context are critical to address new features in future versions in earnest, and delivering what the OpenAPI, Arazzo, and Overlay communities really need.</p>
