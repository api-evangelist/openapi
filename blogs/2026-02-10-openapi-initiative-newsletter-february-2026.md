<p>Welcome to the OpenAPI Initiative (OAI) February 2026 newsletter!</p>
<p>This is our first newsletter of 2026, and there is already plenty to share from the OAI community.</p>
<h3 id="initiative-news">Initiative News</h3>
<p>We’ve made a great start to 2026 in the Overlay world with completion of version 1.1.0 of the <a href="https://spec.openapis.org/overlay/v1.1.0.html">Overlay Specification</a>. Version 1.1.0 brings several new features, including a new <code>copy</code> property for the Action Object, which can be used to copy or move an element in the OpenAPI document. This property is great for operations where you may already have a source-of-truth embedded in part of your API description, and want to use it across all operations. For example, you might implement a <code>default</code> Response Object and wish to copy it to all Operation responses to give more consistency for API consumers.</p>
<p>Other improvements include the ability to <code>update</code> primitive values &#8211; strings, integers, and so on &#8211; rather than updating the parent object, making Overlay documents more concise and easier to manage. The specification has also been updated for full compliance with <a href="https://www.rfc-editor.org/rfc/rfc9535">RFC 9535</a>, to ensure tooling makers are creating Overlay tools through guidance that is fully compatible with the latest JSONPath standards.</p>
<p>To learn more about upgrading, use this <a href="https://learn.openapis.org/upgrading/overlay-v1.0-to-v1.1.html">helpful guide</a>. As always, new versions of our specifications are nothing without our community of contributors, so a <em><em>huge</em></em> thanks to <a href="https://www.linkedin.com/in/vincentbiret/">Vincent Biret</a>, <a href="https://www.linkedin.com/in/lornajane/">Lorna Mitchell</a>, <a href="https://www.linkedin.com/in/ralfhandl/">Ralf Handl</a>, and <a href="https://www.linkedin.com/in/michael-kistler/">Michael Kistler</a> for contributions, reviews, and support.</p>
<p>The <a href="https://github.com/OAI/sig-moonwalk/discussions">Moonwalk Special Interest Group</a> (SIG) has also entered 2026 with a new focus. The SIG will continue to explore concepts that go beyond the current OpenAPI v3 Specification, with a particular focus in the first six months of 2026 on how OpenAPI relates to large language models (LLMs) as a new class of API clients. The groups aims to investigate what additional metadata or structural information might be needed in OpenAPI documents to make them more “agent-ready” for LLM use, including capability discovery and intent signaling. Several open questions are being posed, including around surfacing capabilities, grouping functionality for agents, and optimizing descriptions for LLM-based workflows. You can find the full scope of this initiative <a href="https://github.com/OAI/sig-moonwalk/discussions/219">here</a>.</p>
<p>As always, initiatives as important as this do not create themselves (whatever the capabilities of AI), so the Moonwalk SIG is looking for new contributors and hosts to help in this important work. The Moonwalk SIG is every Tuesday at 1700 GMT / 0900 PST, with the agenda for each meeting published on the GitHub Discussions link above.</p>
<p>Building on the success of v3.2, and the continued great feedback we are getting from the community, we are looking to enhance our coverage of important API security specification in future versions, including the <a href="https://openid.net/wg/fapi/">FAPI Security Profile</a>, which is commonly used in open banking and open finance, and <a href="https://openid.net/wg/authzen/">AuthZEN</a>. We also have the <a href="https://github.com/OAI/sig-industry-standards">Industry Standards Special Interest Group</a>, which looks specifically at industry collaboration and how OAI specifications can meet the needs of API providers across a range of verticals. If you are interested in taking part the Industry Standards SIG to help foster collaboration across industries join the channel in <a href="https://open-api.slack.com/archives/C05UCEVL7M1">Slack</a>. You can also read the draft work plan for 2026 <a href="https://docs.google.com/document/d/1jAVdH_3D94VbNYDM06OvCl0TEUYyeurN_pHhGAp3THs/edit?usp=sharing">here</a> to learn more about the goals of the SIG. SIG meetings are on Mondays at 1730 GMT / 0930 PST and are bi-weekly.</p>
<p>2026 promises to be another exciting year for OAI, with the growing opportunity of specification updates, collaboration with industry verticals and software foundations who rely on OAI specifications, and the exploration into greater compatibility with AI and agentic tooling. We are always on the lookout for new members, so if you are thinking of getting involved, becoming a member is one way of contributing to the work OAI does if you are unable to contribute through specification maintenance or taking part in SIGs. Head over to the <a href="https://www.openapis.org/membershipjoin">membership page</a> on our website to find out more.</p>

<h3 id="ecosystem-spotlight-jentic-ai-readiness-scorecard">Ecosystem Spotlight: Jentic AI-Readiness Scorecard</h3>
<p>Our Ecosystem Spotlight focuses on the work of OAI members and the OAI community in general in using OAI specifications in tooling, products, and experiences. The Ecosystem Spotlight in this newsletter is provided by <a href="https://jentic.com">Jentic</a>, an OAI member who leverages both OpenAPI and Arazzo to provide deterministic and reliable agentic workflows.</p>
<p><em>As AI agents become first-class API consumers, a question emerges that linters can’t answer: can an agent actually reason about this API, and use it safely?</em></p>
<p><em>Jentic’s AI-Readiness Scorecard addresses the gap between specification validity and machine usability. A syntactically correct OpenAPI document guarantees grammar conformance, not that an agent can interpret intent, construct valid requests, or handle errors gracefully. Developers compensate for ambiguity through trial and error; agents either halt or worse, proceed with confident but incorrect assumptions.</em></p>
<p><em>The scorecard evaluates APIs across six dimensions: foundational compliance, developer experience, AI interpretability, agent usability, security, and discoverability. Each dimension produces rich diagnostics that pinpoint exactly where improvements yield the highest gains for both human developers and AI agents. Analysis of 1,500+ APIs revealed consistent obstacles: missing server definitions, authentication buried in prose rather than the spec, sparse or contradictory examples, and broken schema references.</em></p>
<figure>
  <img alt="Jentic AI Scorecard Display showing dials in results" src="https://www.openapis.org/wp-content/uploads/sites/31/2026/02/api-quality-score-LH-Public-API-2026-01-29.png" /><figcaption style="text-align: center; margin-bottom: 15px;">Jentic AI-Readiness Scorecard</figcaption></figure>
<p><em>“AI systems don’t just scan API descriptions; they must interpret, reason, and act on them reliably,” notes Frank Kilcommins, Head of Enterprise Architecture at Jentic and co-author of the Arazzo Specification. “The scorecard provides concrete benchmarks and rich diagnostics on score breakdowns, so you know where you are and what investments will return the most reward. We’re building automatic improvement capabilities leveraging the Overlay specification, helping teams move from insights to actionable fixes.”</em></p>
<p><em>You can learn more about the scoring framework, and/or try it for free with your own APIs at <a href="https://jentic.com/scorecard">https://jentic.com/scorecard</a>.</em></p>
